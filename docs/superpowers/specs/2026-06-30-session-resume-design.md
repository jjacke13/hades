# Session resume — design spec

**Date:** 2026-06-30
**Status:** approved (brainstorm via Q&A; design confirmed "all good"). Ready for plan.

## Goal

A hades restart (REPL or `--serve`) must not lose the conversation. Persist the turn-by-turn chat per
session to disk and reload it on `--resume`, so the agent regains full context. The full history always
persists; only a bounded recent window is sent to the LLM each turn (so a long session can't overflow the
model's context). Separate files per session set up a future embeddings corpus.

## Current state (what exists)

- The **Arbiter** holds `std::vector<nlohmann::json> history_` — the clean turn-message list: `{role:user,
  content}`, `{role:assistant, content|tool_calls}`, `{role:tool, tool_call_id, content}`. In-memory only.
  The system prompt + retrieved memory are injected per-turn and are **never** stored in `history_`, so it
  is exactly the conversation and nothing else. Pushed in 4 places (`arbiter.cpp`: user msg, assistant msg,
  tool result, approved-pending).
- The **Eventlog** appends every bus Entry to `session.log` (TSV, redacted, append-only) — an audit/replay
  trail (`hades-scope`), NOT a clean message list. Resume does NOT reconstruct from it (lossy/fiddly).
- Two **memory layers** (core `memory/facts.md`, archival `.hades/memory.jsonl`) persist separately; this
  feature is orthogonal — the turn-by-turn conversation.
- The Arbiter has setter-style config hooks (`set_tools/set_model/set_system_prompt/set_memory_path`).

## Components

### 1. Per-session store — `.hades/sessions/<id>.jsonl`
- `id` = launch timestamp string (e.g. `20260630-221544`), generated by the binary at startup for a NEW
  session (the C++ binary may read the real clock; only workflow-scripts forbid it). Collision-safe enough
  for human-paced launches; if a file with that id exists, append a `-N` suffix.
- One JSON object per line = one `history_` message, in the EXACT format the Arbiter stores (so reload is a
  verbatim round-trip — no transform).
- **Append-per-message:** add `void append_history(const nlohmann::json& msg)` to the Arbiter that does
  `history_.push_back(msg)` AND, if a session path is set, appends `msg.dump() + "\n"` to the file. Replace
  the 4 raw `history_.push_back(...)` sites with `append_history(...)`. Crash-safe (each message durably on
  disk as it happens), append-only (matches the Eventlog/memory ethos).
- Config: `Session { sessions_dir = .hades/sessions }` (default `.hades/sessions`; store paths must be
  whitespace-free per existing wiring rules). `.hades/` is already gitignored. The parent dir is created if
  absent (like `pin_fact`'s parent-dir creation).

### 2. Session selection (CLI) — `--resume [id]`
`hades_main` argv handling (alongside `--serve`):
- **no `--resume`** → NEW session: generate `id`, create `.hades/sessions/<id>.jsonl`, empty `history_`.
- **`--resume`** (no id) → resume the MOST-RECENT session file in `sessions_dir` (by lexical-max filename =
  newest timestamp; fall back to mtime). If none exists → start a new session + a one-line note to stderr.
- **`--resume <id>`** → resume `.hades/sessions/<id>.jsonl`; if missing → `MalConfig`/clear error (the user
  named a session that isn't there).
- The resolved session path is passed to the Arbiter via wiring (`set_session_path`).
- `--resume` composes with `--serve` (resume then serve) and with the REPL.

### 3. Load on resume + the per-turn overflow guard
- `Arbiter::load_history()` (called once, after `set_session_path`, before the first turn): read the jsonl,
  parse each line tolerantly (skip a blank/corrupt **trailing** line — a crash mid-append — like
  `load_memories`), populate `history_`. A malformed interior line is skipped with a warning (best-effort;
  do not abort the whole resume).
- **Overflow guard in `start_turn()`** (the LLM-request builder): today it copies all of `history_` into the
  request `messages`. Change to include the full history UNLESS its serialized size exceeds
  `history_budget_chars` (Session block, default **120000** ≈ ~30k tokens) — then include only the most-recent
  suffix of messages whose cumulative `dump()` size ≤ budget, dropping oldest. **Pairing invariant:** never
  start the window on a `{role:tool}` message orphaned from its `{role:assistant, tool_calls}` (a tool result
  with no preceding assistant tool_call is invalid to most providers) — if the cut lands mid-pair, drop the
  whole leading partial group so the window begins on a `user` or a complete assistant turn. The system-prompt
  message and the ephemeral retrieved-memory block are added as today (outside the window count; they're not
  in `history_`). Setting `history_budget_chars` very high ⇒ effectively "always send the full session."
  - This guard also fixes a PRE-EXISTING latent bug: `history_` already grows unbounded within one long live
    session today (no resume needed) and would eventually overflow the context — the window bounds it.

### 4. `/new` REPL command
- In `ChatModule::run_repl`/`run_repl_readline`, intercept `/new` (alongside `/quit`): post a `NEW_SESSION`
  bus message (or call a wired hook) → the Arbiter clears `history_`, the binary rotates to a fresh session
  id + file (new `set_session_path`). v1: REPL only. (HTTP `/new` endpoint deferred.) Keep it minimal: if a
  clean bus-decoupled path is awkward, a small wired callback from ChatModule → a "session manager" seam is
  acceptable; prefer a bus message for consistency.

### 5. Web UI (v1) — silent context, blank page
- On `--serve --resume`, the Arbiter loads history → the LLM has full context, but the browser page starts
  blank as today (it does not fetch/render prior turns). No web change this round. `GET /history` +
  page-render is an explicit follow-up.

## Data flow

```
launch: hades <manifest> [--resume [id]] [--serve]
  resolve session path: new id  OR  newest/<id> file        (hades_main)
    → wire: arbiter.set_session_path(path); arbiter.load_history()   (resume only)
turn:
  USER_MESSAGE → append_history(user msg)  → push_back + append file
  start_turn() → LLM_REQUEST{ system + memory + WINDOW(history_, budget) }   ← overflow guard
  LLM_RESPONSE → append_history(assistant msg) ; (tool) TOOL_RESULT → append_history(tool msg) → loop
exit/restart → next launch with --resume reloads the same jsonl verbatim
```

## Error handling

- Missing `sessions_dir` → create it (recursive). Permission/IO error on append → log once to stderr, keep
  running in-memory (don't crash the turn over a disk hiccup; resume just won't have the latest).
- Corrupt trailing line on load → skip (mid-append crash). Corrupt interior line → skip + warn.
- `--resume <id>` not found → clear error/exit. `--resume` with empty `sessions_dir` → new session + note.
- The session file is conversation, not secrets (the api key never enters `history_`); it lives under the
  gitignored `.hades/`. Tool results may contain file/fetch content — same trust boundary as the Eventlog.

## Testing (TDD)

- Arbiter: `append_history` pushes to `history_` AND appends a line when a path is set (and ONLY pushes when
  no path — the test overload). `load_history` round-trips a written jsonl into `history_` verbatim; tolerates
  a corrupt trailing line. The overflow window: a `history_` exceeding the budget yields a request whose
  messages are the recent suffix within budget AND begins on a valid (non-orphan-tool) boundary; a small
  history is sent whole.
- Wiring/main: a `Session { sessions_dir=… }` resolves; `--resume <id>` loads that file; no-flag creates a
  new id; `--resume` newest-file selection. (Unit-test the path-resolution helper; keep the clock injectable
  or test the resolver with a fixed id.)
- Chat: `/new` clears history + rotates the session file.
- Existing 175 tests stay green: the Arbiter test overload sets NO session path → `append_history` is
  push-only → behavior identical; the overflow guard is a no-op below budget (default 120000) so existing
  conversations are sent whole.

## Out of scope (noted)

- `GET /history` endpoint + web UI re-render of prior turns (the natural next step for `--serve`).
- Embeddings / summarization over the session files (the v2 the separate-files design enables; pairs with the
  archival-memory `rank_memories` seam).
- REPL `/sessions` list + `/session <id>` switch; multi-session in the web UI; pruning/retention of old
  session files.

## MOOS-IvP framing

The session store is the **mission log made resumable**: the Eventlog (`.alog`) stays the full audit trail,
while the per-session message file is the helm's working memory of the current mission, reloaded on restart
so the community comes back up where it left off — and the directory of past missions becomes the corpus a
future retrieval behavior can learn from.
